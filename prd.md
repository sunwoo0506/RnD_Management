# 과제온 — 한 페이지 PRD (MVP)

**버전:** v2.2 | **작성일:** 2026-07-16 | **상태:** Draft

> v2.2 변경: 기술 스택을 React 단일 스택으로 확정하고 §6 기술 스택 섹션 신설 (Streamlit 병행 구현 제거, 배포는 Vercel 정적 호스팅). 저장 전략은 §5.1 "로컬 MVP → 서버 베타 → SaaS" 3단계 로드맵 참조.

---

## 1. 만들려는 이유

정부 R&D 과제를 수행하는 1~5인 스타트업에서는 전담 경영관리 인력이 없다. 대표 또는 겸직 담당자가 엑셀과 카카오톡 질문방에 의존해 예산 편성·집행·증빙·정산·변경을 처리하고 있으며, 이 과정에서 참여율 초과, 증빙 누락, 비목 비율 초과 등의 실수가 반복되고 있다.

커뮤니티 채팅 데이터 1,490건 분석 결과, 인건비/참여율 질문이 169건으로 1위, "변경" 키워드가 1,536회로 전체 최다, "불인정" 213회로 나타났다. 이 실수들은 사후에 발견되어 정산 불인정(=기업 자체 부담)으로 이어진다.

과제온은 **각 단계에서 실수를 사전에 차단하는 가이드형 SaaS**를 만들어, "처음 하는 사람이 따라만 하면 되는" 도구를 제공한다.

## 3. 핵심 사용 장면

**장면 A: 예산 편성 (과제 선정 직후)**
담당자가 중기부 과제의 총 사업비를 입력하면, 비목별 허용 비율이 자동 표시되고 배분 초안이 생성된다. 금액을 조정할 때 비율을 초과하면 즉시 경고가 뜬다. 확정 후 엑셀로 내보낸다.

**장면 B: 집행 & 증빙 관리 (과제 수행 중, 매일~매주)**
담당자가 집행건을 등록하면 비목별 잔액이 즉시 갱신되고, 중기부 기준의 필요 증빙 체크리스트가 자동으로 뜬다. 참여율 합산이 100%를 초과하면 빨간 경고가 표시된다. 마감 D-30/14/7에 증빙 누락 알림 이메일이 발송된다.

**장면 C: 예산 변경 (과제 수행 중, 필요 시)**
담당자가 비목 간 금액 이동을 입력하면, 변경 전/후 비교표가 자동 생성된다. 상황에 맞는 변경 사유 문구 템플릿이 제공되고, 중기부 양식의 공문 초안이 자동 생성된다. Word/PDF로 내보낸다.

**오류 시나리오:**
- 집행 금액이 비목 잔액을 초과하는 경우 → 등록 차단이 아닌 **경고 표시 후 사용자 확인 시 등록 허용** (실무에서 일시적 초과 후 변경 신청하는 경우가 있음)
- 참여율 합산 데이터를 입력하지 않은 경우 → 경고 기능이 작동하지 않음을 안내하고 입력 유도
- 증빙 파일 업로드 실패 시 → 재시도 안내, 체크리스트에 "미완료" 상태 유지

## 5. 저장할 데이터

### 5.1 저장 전략과 단계

현재 MVP는 서버 없이 제품 흐름과 규칙을 빠르게 검증하는 단계다. 구조의 최종 목표는 서버 DB 기반 SaaS이며, 브라우저 저장소는 영구 운영 아키텍처가 아니라 로컬 MVP용 구현으로 한정한다.

| 단계 | 저장 방식 | 적용 범위 | 제약 및 목적 |
|------|-----------|-----------|--------------|
| 현재: 로컬 MVP | 프로젝트 데이터는 `localStorage`, 증빙 원본은 `IndexedDB` | 한 브라우저의 한 사용자 | 새로고침 후 데이터 유지와 핵심 사용자 흐름 검증. 브라우저·기기 간 공유, 계정, 백업, 동시 편집은 지원하지 않음 |
| 다음: 서버 베타 | 관계형 서버 DB + 객체 스토리지 | 인증된 사용자와 소속 기업 | 여러 기기 접근, 계정별 데이터 격리, 백업, 이메일 알림, 2인 협업 지원 |
| 운영: SaaS | 관리형 DB + 객체 스토리지 + 감사/백업 체계 | 다중 기업·다중 과제 | 가용성, 보안, 모니터링, 데이터 보존·삭제 정책 충족 |

서버 전환 시 구조를 전면 재작성하지 않도록 다음 원칙을 적용한다.

- UI 컴포넌트가 `localStorage`나 `IndexedDB`를 직접 호출하지 않고 데이터 저장 인터페이스(Repository/Storage Adapter)를 통해 조회·저장한다.
- 현재 로컬 어댑터와 향후 서버 API 어댑터가 동일한 도메인 모델과 메서드 계약을 사용한다.
- 모든 주요 엔터티는 서버 DB에서도 유지 가능한 고유 ID와 생성 시각을 가진다. 서버 전환 전 수정 시각(`updatedAt`)과 스키마 버전(`schemaVersion`)을 추가한다.
- 증빙 메타데이터와 파일 원본을 분리한다. 서버 전환 후 메타데이터는 DB, 파일 원본은 객체 스토리지에 저장하고 DB에는 파일 키·크기·형식·업로드 상태만 기록한다.
- 날짜·시간은 서버 전환 시 UTC ISO 8601로 저장하고 화면에서 사용자 시간대로 표시한다. 금액은 정수 원 단위로 저장한다.
- 브라우저 저장 데이터에는 비밀번호, 인증 토큰, 민감한 서버 비밀값을 저장하지 않는다.

### 5.2 데이터 항목

| 데이터 | 항목 | 비고 |
|--------|------|------|
| 계정 | 이메일, 비밀번호(해시), 이름, 소속 기업명 | 서버 DB 전환 단계에서 도입, 계정당 최대 2명 |
| 과제 | 과제명, 총 사업비, 과제 기간(시작/종료), 정산 마감일 | 스타터 1개, 프로 3개 |
| 참여 인력 | 이름, 참여율(%), 타 과제 참여율(수동 입력) | 참여율 합산 경고용 |
| 비목별 예산 | 비목명, 편성 금액, 허용 비율 상한 | 중기부 규정 데이터 내장 |
| 집행 내역 | 일자, 비목, 금액, 용도, 거래처 | 건별 등록 |
| 증빙 | 비목별 필요 서류 목록, 파일 메타데이터, 파일(업로드), 완료 여부 | 현재 원본은 IndexedDB, 서버 전환 후 객체 스토리지. 건당 최대 10MB, 과제당 500MB |
| 변경 이력 | 변경 전/후 금액, 변경 사유, 생성일 | MVP에서는 최신 1건만 저장 |
| 중기부 규정 데이터 | 비목별 허용 비율, 비목별 필요 증빙 목록, 항목별 인정 기준 | 시스템 데이터, 사용자 입력 아님 |

**개인정보 관련:**
- 현재 로컬 MVP 데이터는 사용자의 브라우저 프로필에만 저장되며 서버로 전송하지 않는다.
- 서버 베타 이후 수집하는 개인정보: 이메일, 이름, 소속 기업명
- 참여 인력 이름은 과제 내부에서만 사용, 외부 노출 없음
- 개인정보처리방침 고지 필수
- 계정 삭제 시 관련 데이터 전체 삭제 (30일 유예 후)

### 5.3 서버 DB 전환 요구사항

서버 저장 기능은 별도 단계에서 구현하되 다음 조건을 만족해야 한다.

| 영역 | 요구사항 |
|------|----------|
| API | 화면에서 저장소 구현을 알 필요가 없도록 과제·예산·집행·증빙·변경 이력 단위의 API 제공 |
| 데이터 격리 | 모든 사용자 데이터는 기업/계정 소유 범위로 조회·수정 권한을 제한 |
| 동시성 | 레코드 버전 또는 `updatedAt`을 이용해 오래된 데이터가 최신 변경을 덮어쓰는 상황 감지 |
| 파일 저장 | 파일은 객체 스토리지에 저장하고 DB에는 메타데이터와 접근 키만 저장. 업로드 완료 후 증빙 상태 확정 |
| 마이그레이션 | 사용자의 명시적 동의를 받은 후 로컬 데이터를 서버로 1회 가져오기. ID 관계 유지, 중복 실행 방지, 성공 확인 전 로컬 원본 보존 |
| 장애 처리 | 네트워크 실패 시 사용자 입력을 즉시 잃지 않도록 재시도 가능한 임시 상태와 명확한 오류 표시 제공 |
| 보안 | TLS, 비밀번호 단방향 해시, 최소 권한 접근 제어, 중요 변경 감사 로그 적용 |
| 백업/삭제 | 자동 백업과 복구 절차를 마련하고 계정 삭제·보존 기간 정책을 구현 |

## 6. 기술 스택

로컬 MVP는 서버 없이 정적 웹앱으로 완결하고, 서버 베타 단계에서 백엔드를 추가한다. 두 벌로 병행하던 Streamlit 구현(app.py)은 제거하고 **React 단일 스택**으로 확정한다.

### 6.1 현재 (로컬 MVP)

| 영역 | 선택 | 비고 |
|------|------|------|
| 프론트엔드 | React + TypeScript + Vite | SPA, 서버 불필요 |
| UI | 자체 CSS (`src/styles.css`), lucide-react 아이콘 | 별도 UI 프레임워크 없음 |
| 데이터 저장 | `localStorage`(프로젝트 데이터) + `IndexedDB`(증빙 원본) | UI는 저장 어댑터(`src/storage.ts`)만 호출 — §5.1 원칙 |
| 문서 내보내기 | write-excel-file(xlsx), docx(Word), jspdf(PDF), file-saver | 브라우저에서 직접 생성, 서버 불필요 |
| 테스트 | Vitest + Testing Library | `npm run test` |
| 배포 | **Vercel 정적 호스팅 (무료 Hobby)** | GitHub 연동 시 `git push`마다 자동 배포. 상업 서비스 전환 시 Cloudflare 이전 검토 (무료 상업 사용·트래픽 무제한) |

### 6.2 서버 베타 전환 시 (예정)

| 영역 | 방향 | 비고 |
|------|------|------|
| 백엔드/DB | 관리형 관계형 DB (예: Supabase — PostgreSQL + 인증 + 스토리지 일체형) | §5.3 서버 DB 전환 요구사항 충족이 선정 기준 |
| 인증 | 이메일 로그인, 계정당 최대 2명 | 서버 베타 단계에서 도입 |
| 파일 저장 | 객체 스토리지 (DB에는 메타데이터만) | §5.1 파일 분리 원칙 |
| 이메일 알림 | 발송 서비스 연동 (D-30/14/7) | 발송 로그 필수 (§8 T-08) |
| 데이터 이전 | 로컬 데이터 1회 가져오기 | §5.3 마이그레이션 요구사항 |

프론트엔드는 저장 어댑터를 로컬 구현에서 서버 API 구현으로 교체하는 방식으로 전환하며, 화면·업무 로직은 재작성하지 않는 것을 목표로 한다.

## 7. 제외 범위

| 제외 항목 | 사유 |
|-----------|------|
| 중기부 외 기관 (산업부, IITP, 과기부) | MVP는 중기부 규정만 내장. 다른 기관은 고객 요청 기반으로 추가 |

| 결제 연동 (PG) | MVP 범위 밖. 계좌이체 수동 결제로 운영 |
| 모바일 앱 | 웹 먼저, PMF 확인 후 |
| 팀 협업 (3인 이상) | 계정당 2명까지만 지원 |
| OCR 증빙 자동 인식 | 2단계 |
| 연도별 분할 편성 | 2단계 |
| 변경 이력 버전 관리 | MVP에서는 최신 1건만 저장 |
| 카카오 알림톡 | 이메일 알림만 지원 |
| 서버 DB·계정·다중 기기 동기화 | 현재 로컬 MVP에서는 제외하며 서버 베타 단계에서 도입 |

## 8. 완료 기준

아래 항목을 **모두 통과**해야 MVP 완료로 판정한다.

### 기능 테스트 (직접 실행 가능)

| # | 테스트 | 통과 조건 |
|---|--------|-----------|
| T-01 | 과제를 등록하고 총 사업비 1억 원을 입력한다 | 중기부 비목별 허용 비율이 화면에 표시되고, 배분 초안이 자동 생성된다 |
| T-02 | 인건비를 총 사업비의 60%로 조정한다 | 빨간 경고가 즉시 표시된다 (중기부 인건비 상한 50% 기준) |
| T-03 | 편성 확정 후 엑셀 내보내기를 누른다 | xlsx 파일이 다운로드되고, 열었을 때 비목·금액이 정확하다 |
| T-04 | 집행건을 등록한다 (회의비 8만 원) | 회의비 잔액이 8만 원 차감되어 표시되고, 중기부 회의비 증빙 체크리스트가 자동으로 뜬다 |
| T-05 | 비목 잔액보다 큰 금액의 집행건을 등록한다 | 경고가 표시되고, 사용자 확인 후 등록이 가능하다 |
| T-06 | 증빙 체크리스트에 파일을 업로드한다 | 파일이 저장되고 해당 항목이 "완료"로 체크된다 |
| T-07 | 참여 인력의 타 과제 참여율을 입력하여 합산 110%로 만든다 | 빨간 경고가 즉시 표시되고, 100% 이하로 조정하라는 안내가 나온다 |
| T-08 | 정산 마감일을 30일 뒤로 설정하고 증빙 미완료 건을 남긴다 | D-30 알림 이메일이 발송된다 (이메일 발송 로그로 확인) |
| T-09 | 비목 간 금액 이동을 입력한다 | 변경 전/후 비교표가 자동 생성되고, Word 또는 PDF로 다운로드된다 |
| T-10 | 변경 사유 템플릿을 선택하고 공문 초안을 생성한다 | 중기부 양식의 공문이 생성되고, Word로 다운로드된다 |
| T-11 | 1계정에 2명을 등록한다 | 두 번째 사용자가 같은 과제 데이터에 접근할 수 있다 |
| T-12 | 과제 등록 → 편성 → 집행 5건 → 변경 1건을 연속으로 수행한다 | 전체 흐름이 끊김 없이 완료되고, 데이터가 모듈 간 일관성 있게 연결된다 |
| T-13 | 데이터를 입력한 뒤 같은 브라우저에서 페이지를 새로고침한다 | 과제·예산·집행·변경 데이터와 증빙 상태가 그대로 복원된다 |
| T-14 | UI 코드의 저장 기능을 점검한다 | UI가 브라우저 저장소를 직접 호출하지 않고 공통 저장 인터페이스를 통해 동작한다 |

T-08(실제 이메일 발송)과 T-11(2인 공동 접근)은 서버 DB·인증·백엔드가 도입되는 서버 베타 단계의 완료 기준이다. 현재 로컬 MVP에서는 이메일 화면/로그 흐름과 단일 사용자 핵심 업무 흐름만 검증한다.

### 성공 지표와 측정 방법

| 지표 | 측정 방법 | 목표 | 비고 |
|------|-----------|------|------|
| 셀프 온보딩 완료율 | 가입 후 과제 등록까지 완료한 사용자 비율 | [가정] 70% 이상 | "가이드형 SaaS"의 본질 검증 |
| 증빙 완비율 | 등록된 집행건 중 증빙 체크리스트 100% 완료 비율 | [가정] 80% 이상 | 베타 운영 후 기준선 확정 |
| 참여율 경고 대응률 | 경고 발생 후 참여율 조정을 완료한 비율 | [가정] 90% 이상 | 경고 로그 + 이후 수정 여부로 측정 |
| 주간 활성 사용률 | 최근 7일 내 로그인한 사용자 / 전체 등록 사용자 | [가정] 60% 이상 | 정산 시즌(11~12월) 별도 분리 측정 |
| 베타 테스터 계속 사용 의향 | "실제 과제에 계속 쓰겠다" 응답 비율 (설문) | 베타 10명 중 5명 이상 | PMF 신호 |

### 운영 리스크

| 리스크 | 영향 | 대응 |
|--------|------|------|
| 중기부 규정이 연도 중 변경됨 | 과제온의 비율/증빙 안내가 구버전이 되어 사용자가 잘못된 정보로 집행 | 규정 데이터에 "기준일" 표시. 규정 변경 모니터링 프로세스 수립 (수동, 월 1회). 변경 시 사용자 알림 |
| 증빙 파일 업로드 중 데이터 유실 | 사용자가 올린 증빙 파일이 사라짐 | 업로드 완료 확인 응답 후 체크 처리. 일일 백업 |

| 개인정보 유출 | 이메일, 이름, 기업명 노출 | TLS 전송 암호화, 저장 시 AES-256, 접근 제어(계정별 데이터 격리), 개인정보처리방침 고지 |
| 브라우저 데이터 삭제·기기 변경 | 현재 로컬 MVP 데이터가 복구되지 않음 | 로컬 저장 한계를 화면에 고지하고 내보내기/백업 기능 검토. 서버 베타 전환 시 로컬 데이터 가져오기 제공 |
| 로컬 저장 구조가 UI에 결합됨 | 서버 DB 전환 시 화면과 업무 로직까지 대규모 재작성 | 저장 인터페이스와 로컬/서버 어댑터를 분리하고 계약 테스트로 두 구현의 동작 일치 검증 |
