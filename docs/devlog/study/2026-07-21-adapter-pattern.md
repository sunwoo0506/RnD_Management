# [스터디] 어댑터 패턴 — 형식이 다른 데이터를 우리 형식으로 바꿔 쓰는 변환기 (2026-07-21)

## 무엇을 공부했나
외부에서 만들어진 데이터 묶음(지침 추출 프레임워크의 규정 DB)을 우리 앱이 쓰는 형식(규정팩)으로 바꿔주는 "어댑터(변환기)"를 만드는 방법을 공부했습니다.

## 왜 필요했나
지침을 자동으로 DB화하는 프레임워크가 만든 결과물(비목 트리, 상한 규칙, 사용 항목 등 JSON 파일 묶음)은 우리 앱의 규정팩 구조와 이름도 모양도 달랐습니다. 이걸 앱에서 쓰려면 두 가지 길이 있었는데 — ① 앱 전체를 그 구조에 맞게 뜯어고치기, ② 중간에 변환기 하나 두기 — ①은 몇 주짜리 공사고 ②는 하루면 됐습니다.

## 쉽게 설명하면
해외여행용 "돼지코"(전원 플러그 변환 어댑터)와 같습니다. 한국 가전제품(우리 앱)을 유럽 콘센트(외부 데이터 형식)에 꽂으려고 제품의 플러그를 잘라 갈아 끼우지 않습니다. 돼지코 하나만 챙기면 제품은 그대로 쓰고, 나라(데이터 형식)가 바뀔 때마다 돼지코만 바꾸면 됩니다.

우리 경우 돼지코가 `scripts/convert-regulation-db.mjs`입니다. 예를 들어:
- 외부 형식: `{"limit_type": "PERCENT", "limit_value": 20, "basis_code": "modified_labor_cost"}`
- 우리 형식: `{"kind": "ratio", "limitPct": 20, "basis": "수정인건비 합계"}`

변환기가 이런 대응표를 갖고 자동으로 옮겨 적습니다.

## 실제로 어떻게 적용했나
1. 국가연구개발비 사용기준 2026 패키지 → 명령어 한 번으로 영리·비영리 규정팩 2개 생성.
2. 팁스(TIPS) 운영지침 패키지 → 같은 명령어로 팁스 팩 생성. 앱 코드는 "팩 목록에 한 줄 추가"밖에 안 고쳤습니다.
3. 패키지 안내파일(manifest)에 팩 이름·기관·출력 파일명을 적게 해서(`pack_meta`), 앞으로 어떤 지침이든 변환기 수정 없이 처리되게 했습니다.
4. 다만 "편성 화면에서 계산할 수 없는 규칙"(예: 건별 5만원 한도)은 상한이 아니라 주의사항으로 분류하는 규칙도 대응표에 넣었습니다 — 변환은 기계적 복사가 아니라 "우리 화면에서 어떤 의미인가"를 정하는 일이라는 걸 배웠습니다.

## 더 알아볼 것
- 변환기가 놓치는 규칙(계산 불가 기준)을 나중에 서버 판정 엔진으로 처리하는 방법
- 지침이 개정됐을 때(베이스 버전 변경) 그 위에 얹힌 사업 팩들을 자동으로 재생성하는 방법
