# [스터디] PDF는 글자를 "좌표에 찍어둔 그림"에 가깝다 (2026-07-23)

## 무엇을 공부했나

PDF 안에 글자가 어떤 모습으로 저장되는지, 그리고 거기서 문장을 되살릴 때
왜 "그냥 읽으면" 안 되는지를 공부했습니다.

## 왜 필요했나

규정 지침 PDF에서 증빙서류 표를 뽑았더니 문장이 이렇게 나왔습니다.

```
세금계산서 신용카드 영수증( ),  견적서,  검수조서 증빙사진 포함( ), 거래처
```

괄호가 전부 엉뚱한 자리에 몰려 있어 **무슨 서류를 말하는지 읽을 수가 없었습니다.**
이걸 규정DB에 그대로 실으면 사용자가 근거를 눌렀을 때 깨진 글을 보게 됩니다.

## 쉽게 설명하면

워드 파일은 **"이 문장을 썼다"** 를 저장합니다. 문장이 하나의 덩어리로 들어 있습니다.

PDF는 다릅니다. **"이 글자를 이 위치에 찍어라"** 만 저장합니다.
인쇄용 형식이라, 종이 어디에 잉크를 묻힐지가 전부입니다. 문장이라는 개념이 없습니다.

> 워드가 **편지**라면, PDF는 **도장 찍는 순서표**입니다.
> "세금계산서를 왼쪽 154 지점에", "괄호를 209 지점에" 하는 식이죠.

그래서 PDF에서 글을 읽어내는 프로그램은 좌표를 보고 **문장을 다시 조립**해야 합니다.
보통은 "위에서 아래로, 왼쪽에서 오른쪽으로" 이으면 됩니다.

문제는 한글(HWP)에서 PDF로 내보낼 때 생겼습니다. 좌표를 하나씩 찍어보니 이랬습니다.

```
x=154.2  "세금계산서 신용카드 영수증"   ← 본문을 부호 없이 한 덩어리로
x=209.2  "("                          ← 그 덩어리 '위에' 덧그림
x=220.2  ","
x=220.2  " "                          ← 쉼표와 똑같은 자리!
```

본문에는 부호가 들어갈 자리마다 **빈칸을 비워두고**, 부호는 그 위에 따로 도장을 찍은 것입니다.
왼쪽부터 순서대로 줄을 세우면, 같은 자리에 겹쳐 있는 부호는 갈 곳을 잃고 줄 끝으로 밀립니다.

## 실제로 어떻게 적용했나

"부호가 덮은 빈칸이 어디인지"를 계산해서 되돌려놨습니다.

1. 본문 덩어리는 글자 폭이 고르게 짜여 있으니, 부호의 좌표를 **몇 번째 글자인지**로 환산합니다.
   `(209.2 − 154.2) ÷ 덩어리 폭 × 글자 수` → 약 4.6번째 → 5번째 자리
2. 그 자리가 빈칸이면, 거기가 원래 부호가 있던 곳입니다. 빈칸을 부호로 바꿉니다.

```
되돌린 결과: 세금계산서(신용카드 영수증), 견적서, 검수조서(증빙사진 포함), 거래처 사업자등록증
```

**한 번 헛디딘 것도 있습니다.** 처음엔 "부호처럼 생긴 건 다 덧그린 것"으로 봤더니,
표에서 값이 없다는 뜻으로 쓰는 `-` 칸까지 앞 칸에 붙여버렸습니다. 표의 열이 하나 사라진 거죠.
기존 테스트가 이걸 잡아줘서, **"다른 글자 위에 겹쳐 있는 부호만"** 으로 조건을 좁혔습니다.
제 자리를 차지하고 있는 부호는 손대지 않습니다.
