# [스터디] 승인이 필요한 일은 "신청"과 "반영"을 갈라야 한다 (2026-07-24)

## 무엇을 공부했나

바깥의 승인을 받아야 효력이 생기는 일을 앱에서 어떻게 다뤄야 하는지 —
**신청(요청)** 과 **반영(적용)** 을 나누는 이유와, 나누지 않으면 무엇이 무너지는지를 공부했습니다.

## 왜 필요했나

지금까지 예산 변경 관리는 이렇게 동작했습니다.

```
[비목 간 금액 이동] 저장 클릭
        ↓
     예산이 즉시 바뀜
```

그런데 실제 정부 R&D 과제에서 예산 변경은 이렇게 돌아갑니다.

```
① 협약서·사업계획서를 근거로
② 변경사항을 반영한 사업계획서를 만들어 전문기관에 신청
③ 전문기관이 승인하거나 반려
④ 승인된 경우에만 예산이 실제로 바뀜
```

앱이 ①②③을 건너뛰고 ④만 흉내 내고 있었던 것입니다.

## 나누지 않으면 무엇이 무너지나

### 1. 지금 화면의 숫자가 진짜인지 알 수 없다

승인 전에 예산을 바꿔두면, 화면의 "인건비 5,000만원"이
**협약된 금액인지, 내가 바꾸고 싶다고 적어둔 금액인지** 구분되지 않습니다.
집행하다 잔액을 보고 "아직 여유 있네" 했는데 사실 승인이 안 난 금액이라면
그 판단은 통째로 틀린 것입니다.

### 2. 반려됐을 때 되돌릴 근거가 없다

반려는 흔한 일입니다. 그런데 이미 예산을 바꿔놨다면 되돌리려고 할 때
"원래 얼마였지?"를 알아야 합니다. 그 사이 다른 변경이 또 있었다면 더 꼬입니다.

### 3. 왜 못 바꿨는지가 사라진다

반려된 신청을 그냥 지우면 기록이 없어집니다.
몇 달 뒤 같은 변경을 다시 시도할 때 **지난번에 왜 반려됐는지**를 아무도 모릅니다.

## 그래서 어떻게 했나

상태를 네 단계로 두고, **예산이 움직이는 지점을 한 곳으로 못박았습니다.**

| 상태 | 뜻 | 예산은? |
|---|---|---|
| `draft` | 작성 중 | 그대로 |
| `submitted` | 신청함 | **그대로** |
| `approved` | 승인됨 | **이때 바뀐다** |
| `rejected` | 반려됨 | 그대로 (회신 내용만 기록) |

예산이 바뀌는 곳이 `approved` 하나뿐이라, "지금 이 숫자가 왜 이런가"를 되짚을 때
볼 곳이 한 군데입니다.

대신 신청 중인 변경이 안 보이면 안 되므로, **"승인되면 이렇게 된다"는 예상값**을
따로 계산해 함께 보여줍니다. 확정된 값과 예상값은 화면에서 구분됩니다.

## 놓치기 쉬운 함정: 신청과 승인 사이의 시차

처음에는 신청할 때 계산해둔 결과(after)를 승인 때 그대로 적용하려 했습니다.
그런데 이런 일이 생깁니다.

```
9/1  A 신청 (인건비 6,000 → 5,000)   ← 이때 계산한 결과: 5,000
9/2  B 신청 (인건비 6,000 → 5,500)   ← 이때 계산한 결과: 5,500
9/10 A 승인 → 인건비 5,000
9/15 B 승인 → 저장해둔 5,500을 그대로 쓰면?  A의 승인이 사라진다!
```

그래서 **승인 시점에 그때의 예산에서 다시 계산**하도록 했습니다.
저장해둔 숫자를 믿지 않고, "얼마를 어디서 어디로 옮긴다"는 **의도**만 저장해두고
적용은 그때그때 계산합니다.

> 일반화하면 — **결과값이 아니라 연산을 저장한다.**
> 시간이 지나면 결과값은 낡지만 연산은 낡지 않습니다.

## 앞으로 어떻게 쓸까

바깥의 승인·검토가 끼는 일에는 같은 구조를 씁니다.

- 신청과 반영을 나누고, 반영되는 지점을 하나로 못박는다
- 승인 전 상태도 화면에서 보이게 하되 확정된 값과 구분한다
- 반려 기록은 남긴다 — 실패한 시도도 다음 판단의 근거다
- 저장은 결과가 아니라 의도(무엇을 어떻게 바꾼다)로 한다
