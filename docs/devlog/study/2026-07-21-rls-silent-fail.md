# [스터디] RLS(행 수준 보안)가 "조용히 실패"할 수 있다는 것 (2026-07-21)

## 무엇을 공부했나
Supabase 같은 데이터베이스 서비스가 쓰는 RLS(Row Level Security, 행 수준 보안 — "이 사람은 이 데이터를 봐도/고쳐도 되는가"를 데이터베이스가 직접 검사하는 규칙)는, 권한이 없는 요청을 **에러로 막지 않고 그냥 "성공했지만 0건 처리"로 조용히 끝내버릴 수 있다**는 걸 알게 됐습니다.

## 왜 필요했나
사용자가 관리자 화면에서 문서를 삭제했는데, 화면에서는 사라졌지만 실제 데이터베이스에는 그대로 남아있는 문제가 있었습니다. 에러 메시지도 없었고, 화면도 "삭제됐다"고 표시했는데 왜 실제로는 안 지워졌는지 원인을 찾아야 했습니다.

## 쉽게 설명하면
회사 창고에 비유하면 이렇습니다. 창고 관리자 명단에 없는 직원이 "이 상자 좀 치워주세요"라고 요청하면, 보통은 "권한이 없습니다"라고 딱 잘라 거절할 것 같지만, RLS는 그렇게 동작하지 않을 때가 있습니다. 대신 "네, 알겠습니다" 하고 대답은 하는데 실제로는 아무 상자도 안 치우고 그냥 돌아서는 것과 비슷합니다. 요청한 사람 입장에서는 "성공했다"는 응답을 받으니 진짜로 치워진 줄 알지만, 창고엔 그대로 남아있는 거죠.

## 실제로 어떻게 적용했나
이 사실을 알고 나서, 관리자 권한이 필요한 작업(문서 승인·삭제 등)은 **일반 사용자 로그인 권한으로 직접 데이터베이스에 요청을 보내는 방식을 쓰지 않기로** 했습니다. 대신 "Edge Function(서버에서 대신 실행되는 코드)"이라는 별도의 서버 코드를 만들어서, 그 코드 안에서 "이 사람이 진짜 관리자 계정인지"를 먼저 확인하고, 맞으면 데이터베이스에 아무 제약 없이 직접 접근할 수 있는 특수 권한(서비스 롤)으로 작업을 처리하도록 바꿨습니다. 이렇게 하면 권한이 없을 때 "성공은 했는데 아무 일도 안 일어나는" 상황 자체가 애초에 생길 수 없습니다 — 권한 확인에 실패하면 명확하게 에러를 돌려주기 때문입니다.

## 더 알아볼 것
- RLS 규칙을 직접 짤 때, "몇 건이 실제로 처리됐는지"를 항상 확인하는 습관을 어떻게 코드 차원에서 강제할 수 있을지
- Supabase 대시보드에서 RLS 정책이 지금 어떻게 걸려 있는지 한눈에 점검하는 방법
