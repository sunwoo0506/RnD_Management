# [스터디] 디자인 시스템 — 색을 고르지 않고 이름을 부르기 (2026-07-23)

## 무엇을 공부했나

**디자인 시스템(디자인 규칙을 문서로 정리해 둔 것)** 과 **디자인 토큰(색·간격 같은 값에
이름을 붙여 한 곳에 모아둔 것)** 이 무엇이고, 왜 화면마다 색을 직접 고르면 안 되는지를
공부했습니다.

## 왜 필요했나

Stripe(결제 회사)의 디자인 규칙을 정리한 문서를 받아 "이 문서 그대로 앱에 적용해달라"는
요청이 있었습니다. 적용하려고 앱의 스타일 파일을 열어보니 이런 상태였습니다.

```
파랑 계열   #3867e8, #315dc8, #2d57c7, #3b66db, #4267c4, #5479dc …  20가지
회색 계열   #4d586c, #5b667b, #8f98a9, #8490a4, #667087 …          40가지
```

같은 "브랜드 파랑"인데 화면마다 조금씩 다른 값이 쓰이고 있었습니다. 새 화면을 만들 때마다
"어떤 파랑이었지?"를 다시 찾아야 했고, 브랜드 색을 바꾸려면 **200곳 넘게** 손봐야 했습니다.

## 쉽게 설명하면

옷장에 비유하면 이렇습니다.

- **토큰이 없는 상태**: 옷을 살 때마다 "이 정도면 남색이겠지" 하고 고릅니다. 한 벌씩 보면
  괜찮은데, 다 꺼내 놓으면 미묘하게 다른 남색이 스무 벌입니다. 그리고 "우리 집 남색을
  회색으로 바꾸자"는 순간 스무 벌을 전부 다시 사야 합니다.
- **토큰이 있는 상태**: "우리 집 대표색"이라는 **이름표**를 만들고, 옷은 전부 그 이름표를
  가리킵니다. 대표색의 내용물만 바꾸면 스무 벌이 한 번에 따라옵니다.

코드로는 이렇게 생겼습니다.

```css
/* ① 이름을 정의하는 곳 — 한 파일에 한 번만 */
:root {
  --primary: #4a154b;        /* 브랜드 색 */
  --ink: #1d1d1d;            /* 본문 글자색 */
  --line: #e6e6e6;           /* 구분선 */
  --r-pill: 90px;            /* 버튼 둥글기 */
  --shadow-1: rgba(0,0,0,.1) 0 5px 20px;   /* 카드 그림자 */
}

/* ② 쓰는 곳 — 값을 직접 적지 않고 이름을 부른다 */
.primary  { background: var(--primary); border-radius: var(--r-pill); }
.panel    { border: 1px solid var(--line); box-shadow: var(--shadow-1); }
```

## 실제로 겪은 것

이번에 브랜드를 **두 번** 갈아엎었습니다.

1. **1차**: Stripe풍(남색 배경 + 인디고 `#533afd`) 적용
2. **2차**: Slack풍(오버진 `#4a154b` + 크림 배경)으로 교체
3. **중간에** 다른 사이트의 파란 팔레트를 시험 적용했다가 "이전이 낫다"는 판단으로 되돌림

1차에는 색을 하나씩 찾아 바꾸느라 오래 걸렸지만, **토큰을 만들어 둔 뒤인 2차·3차는
훨씬 빨랐습니다.** 되돌리기까지 깔끔하게 됐습니다. 토큰의 값어치는 처음 만들 때가 아니라
**바꿀 때** 나타납니다.

## 규칙 문서에서 배운 것

색뿐 아니라 **"쓰는 방법"까지 규칙**이라는 점이 인상적이었습니다.

| 규칙 | 이유 |
|---|---|
| 브랜드 색은 **화면당 채운 버튼 하나**에만 | 여러 개면 어디를 눌러야 할지 모른다 |
| 브랜드 색을 **본문 글자색으로 쓰지 않는다** | 강조 색은 강조에만 써야 강조가 된다 |
| 금액·숫자는 **고정폭 숫자**로 | 자릿수가 세로로 맞아 비교가 쉽다 |
| 버튼은 **전부 같은 모양**(타원형) | 모양이 섞이면 "이건 버튼인가?"를 매번 판단해야 한다 |

즉 디자인 시스템은 "예쁜 색 목록"이 아니라 **"고민을 줄여주는 결정 모음"** 입니다.
이미 정해진 것은 다시 정하지 않아도 됩니다.

## 앞으로 어떻게 쓸까

- 새 화면을 만들 때 색·간격·둥글기를 **직접 적지 않는다.** 토큰에 없으면 "정말 새 값이
  필요한가"를 먼저 의심한다.
- 토큰에 없는 값이 정말 필요하면 **토큰에 추가**하고 이름을 붙인다. 화면에만 몰래 적으면
  다음 사람이 또 찾아 헤맨다.
- 규칙 문서(`docs/DESIGN-slack.md`)와 실제 코드(`src/styles.css`의 `:root`)가 어긋나지
  않게 유지한다. 문서는 "왜", 코드는 "무엇"을 담는다.

## 관련 파일

- `docs/DESIGN-slack.md` — 현재 적용 중인 규칙 문서
- `src/styles.css` 맨 위 `:root` — 토큰이 정의된 곳
