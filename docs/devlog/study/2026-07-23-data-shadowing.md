# [스터디] 같은 이름의 데이터가 여러 곳에 있을 때 누가 이기나 (2026-07-23)

## 무엇을 공부했나

같은 데이터가 여러 곳에 동시에 존재할 때(앱에 내장된 기본값 · 서버 · 브라우저 임시저장),
프로그램이 어느 것을 쓰는지 — 그리고 그 순서를 모르면 어떤 착각에 빠지는지를 공부했습니다.

## 왜 필요했나

규정 원문을 보여주는 기능을 만들고, 자동 테스트까지 통과했습니다.
그런데 **실제 화면은 하나도 달라지지 않았습니다.**

코드를 아무리 다시 봐도 맞았습니다. 문제는 코드가 아니라 **데이터가 오는 길**이었습니다.

## 쉽게 설명하면

우리 앱에서 규정 데이터는 세 곳에 있습니다.

| 어디 | 무엇 | 언제 쓰이나 |
|---|---|---|
| 앱 안에 내장 | 배포할 때 같이 굽힌 기본값 | 서버에 연결 못 할 때 |
| 서버(Supabase) | 진짜 최신본 | **평소. 같은 이름이면 이쪽이 이김** |
| 브라우저 임시저장 | 서버에서 받아둔 사본 | 화면을 빨리 띄우려고 |

> 도서관에 비유하면 이렇습니다.
> 내 책상 위의 낡은 사본(앱 내장), 도서관 서고의 최신판(서버), 어제 빌려와 가방에 넣어둔 것(임시저장).
> **내 책상 위의 사본을 아무리 고쳐 써도, 남들이 읽는 건 서고에 있는 책입니다.**

이 구조는 일부러 그렇게 만든 것입니다. 규정이 개정될 때마다 앱을 다시 배포하지 않고
서버만 갱신하면 되니까요. 그런데 이번엔 그게 반대로 작용했습니다.
저는 **내 책상 위 사본만 고쳐놓고** "왜 화면이 그대로지?" 하고 있었던 겁니다.

자동 테스트도 같은 함정이었습니다. 테스트는 서버에 연결하지 않고 **앱 내장 데이터**로 돌아갑니다.
그래서 테스트는 초록불인데 사용자 화면은 빨간불인 상태가 될 수 있습니다.

## 실제로 어떻게 적용했나

**추측하지 않고 서버를 직접 조회했습니다.** 이게 핵심이었습니다.

```
tips2026-general    조문  15  증빙표 없음   ← 서버 (2026-07-22 08:55)
tips2026-general    조문  16  증빙표 있음   ← 내 컴퓨터
```

한 줄로 원인이 드러났습니다. 서버에 올린 뒤에는 6개 사업 데이터를
비목·규칙·원문·인정항목 단위로 하나씩 대조해 전부 `동일`인 것을 확인했습니다.

여기서 얻은 습관 하나 — **"고쳤다"와 "사용자에게 닿았다"는 다른 일입니다.**
데이터가 여러 겹으로 쌓인 구조에서는, 고친 뒤에 *어느 겹까지 갔는지* 를 따로 확인해야 합니다.

## 더 알아볼 것

- 브라우저 임시저장이 낡았을 때 자동으로 비우는 조건은 지금 어떻게 되어 있는지
- 서버 데이터가 앱 내장본보다 **오래된** 경우도 있을 수 있는데, 그때 경고를 띄울 수 있을지
