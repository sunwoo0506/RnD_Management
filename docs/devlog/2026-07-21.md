## 🛠 작업 상세

### 1. 공유 문서고 구조 재설계 (문서 · 버전 · 파일 분리)
- **무엇을:** 공고문·지침 같은 규정 파일을 "문서(정체성) · 버전(개정 이력) · 실제 파일" 세 단계로 나눠 저장하도록 데이터베이스(정보를 저장하는 곳)를 새로 설계했습니다. 파일 유형도 법률·시행령·공고문·협약서 등 14가지로 세분화했습니다.
- **왜:** 기존에는 규정이 개정되면 예전 파일을 밀어내고 새 파일로 덮어쓰는 구조라, "지난번 공고문엔 뭐라고 적혀 있었지?"를 나중에 확인할 방법이 없었습니다.
- **결과:** 같은 문서(예: 사업 공고문)에 1차·2차·3차 같은 여러 시행 버전을 계속 쌓아둘 수 있고, 이전 버전은 지워지지 않고 그대로 남습니다.

### 2. 관리자 전용 승인 화면(`/admin`) 신설
- **무엇을:** 일반 사용자가 로그인하는 계정과는 완전히 분리된 별도 화면을 만들었습니다. 사용자가 자료를 올려도 곧바로 공개되지 않고, 이 화면에서 관리자가 확인·승인해야 다른 사용자에게 보이도록 했습니다.
- **왜:** 검증되지 않은 자료가 아무 절차 없이 바로 공개되면, 오타나 잘못된 정보가 섞일 위험이 있어서입니다.
- **결과:** 사용자가 문서를 올리면 "승인 대기" 상태로 들어가고, 관리자가 `/admin`에서 내용을 확인·정리한 뒤 승인해야 모든 사용자에게 공개됩니다.

### 3. 문서 ↔ 사업명 정확한 연결
- **무엇을:** 각 문서를 특정 사업명에 연결하는 기능을 추가했습니다. 관리자 화면에서 사업명을 검색하면, 그 사업에 등록된 문서를 "1차·2차·3차" 버전별로 나무 구조(트리)로 한눈에 확인할 수 있습니다.
- **왜:** 지금까지는 사업명을 사람이 직접 타이핑해서 입력하다 보니, 오타나 띄어쓰기 차이로 같은 사업이 서로 다른 이름으로 여러 번 등록되는 문제가 반복됐습니다.
- **결과:** 같은 사업명이 이미 등록돼 있으면 자동으로 알아채서, 새로 만들지 않고 기존 사업에 연결하도록 화면이 미리 안내해줍니다.

### 4. 사용자 화면 문서 기능 통합
- **무엇을:** "근거 원본 문서", "사업 문서", "공고문 업로드"로 나뉘어 있던 화면 3곳의 문서 관련 기능을 "근거 원본 문서" 하나로 합쳤습니다.
- **왜:** 화면마다 문서를 다루는 방식이 조금씩 달라서 사용자가 헷갈렸고, 같은 기능이 여러 화면에 중복으로 나타나는 문제도 있었습니다.
- **결과:** 이제 문서 업로드 → 공유 신청 → 사업명 검색으로 불러오기 → AI로 규정 분석까지 한 화면에서 자연스럽게 이어집니다.

### 5. (오후) "완료된 규정팩" 보관함
- **무엇을:** AI로 추출해서 적용한 규정팩이 자동으로 보관되고, 과제 설정 → AI 규정 추출 하단에서 언제든 다시 열어볼 수 있게 했습니다. 팝업에서 비목·규칙을 원문 인용과 함께 보고, 근거 링크를 누르면 원문 미리보기 창이 뜨며, 이전 팩을 다시 적용하거나 삭제할 수도 있습니다.
- **왜:** 새 팩을 적용하면 이전 팩이 흔적 없이 사라져서, "지난번 추출 결과가 뭐였더라"를 확인할 방법이 없었습니다.
- **결과:** 최근 20개까지 이력이 쌓이고, 어떤 팩이 현재 적용 중인지 표시됩니다.

### 6. (오후) 계정·과제 안전장치와 다중 과제 지원
- **무엇을:** ① "과제 나가기" 버튼을 실제 동작에 맞게 "과제 삭제"로 바꾸고, 계정을 바꿀 땐 로그아웃을 쓰라고 안내합니다. ② 로그아웃하면 브라우저에 남은 과제 사본을 정리하고, 로그인 없이 만든 데이터만 첫 로그인 때 계정으로 이전되게 했습니다. ③ 한 계정에 과제를 여러 개 등록하고 사이드바에서 전환하는 기능을 만들었습니다(과제별로 클라우드에 저장하는 새 테이블 `user_projects` 추가).
- **왜:** 계정을 바꾸려고 "과제 나가기"를 눌렀다가 과제가 통째로 삭제되는 사고가 있었고, 이전 계정의 데이터가 다음 로그인 계정으로 잘못 복사될 수 있는 구조였습니다. 과제도 계정당 1개만 지원했습니다.
- **결과:** 과제 삭제는 해당 과제 하나만 지우고, 계정 간 데이터 섞임이 차단되며, 사이드바 "현재 과제" 버튼으로 여러 과제를 오가며 작업할 수 있습니다.

### 7. (오후) 예산 편성 화면 개선 3종
- **무엇을:** ① 편성 금액 아래 슬라이더(막대바)로 미편성 잔액 안에서 금액을 끌어서 조절, ② "외부 전문기술 활용비 → 기술도입비·전문가활용비"처럼 비목 안을 세목으로 나눠 입력(비목 금액은 세목 합계로 자동 계산), ③ 비목마다 현물 금액을 입력하면 현금(지원금+민간 현금)이 자동 계산되고, 화면 상단에 "현금 편성/재원 현금"과 "현물 편성/재원 현물" 두 줄로 초과·부족을 검증합니다.
- **왜:** 금액을 일일이 타이핑해야 했고, 세목 단위 편성이나 현금/현물 구분 관리가 아예 없어 공고 요건(현물 한도 등)과 대조할 수 없었습니다.
- **결과:** 재원 구성과 다르게 편성하면 초과/부족 금액이 빨간색으로 바로 표시됩니다. 엑셀 내보내기에도 현금·현물 열이 추가됐습니다.

### 8. (오후) 인건비 관리 신설
- **무엇을:** 인력·담당자 화면에서 참여인력마다 기존/신규 구분, 참여기간, 월급여, 계상 구분(현금/현물/혼합)을 입력하면 4대보험(요율 조정 가능)·퇴직금(월급여의 1/12)·참여율을 반영해 사업기간 합계 인건비를 자동 계산합니다. "예산 편성에 반영" 버튼으로 기존/신규 인건비 합계가 해당 비목에 현금·현물 구분까지 들어갑니다. 4대보험·퇴직금은 사업별 기준이 달라 포함/미포함 체크박스로 선택합니다.
- **왜:** 인건비는 계산식이 정해져 있는데도 사용자가 밖에서 계산해 숫자만 입력해야 했고, 특히 현물 인건비(예: 기존인력)가 예산에 현금으로 잘못 잡히는 문제가 있었습니다.
- **결과:** 사람별 계산 근거가 화면에 그대로 보이고, 인건비 현물 합계가 재원 한도를 넘으면 반영 전에 경고가 뜹니다.

### 9. (오후) 지침 DB화 파이프라인 구축
- **무엇을:** 추출 프레임워크가 만든 규정 DB 패키지(JSON 묶음)를 앱 규정팩으로 바꾸는 변환기(어댑터, `scripts/convert-regulation-db.mjs`)를 만들었습니다. 이걸로 ① 국가연구개발비 사용기준 2026(영리·비영리 2개 팩)과 ② 팁스(TIPS) 총괄 운영지침을 내장 팩으로 등록했습니다. 팁스는 HWP 원본에서 직접 텍스트를 추출해 프레임워크 형식 패키지(비목 19개·규칙 27건, 규칙마다 원문 인용 포함)로 만들었고, 검토용 엑셀을 생성하는 스크립트(`scripts/make_regulation_review.py`)도 만들었습니다.
- **왜:** "예산 규정은 AI가 매번 추출하는 게 아니라 DB에서 가져오고, AI는 미등록 지침을 DB화할 때만 쓴다"는 방향을 정했기 때문입니다. 신규 과제 마법사도 ① DB 검색 → ② 문서 업로드 → ③ AI 추출(DB에 없을 때만) 순서로 재배치했습니다.
- **결과:** 새 지침 패키지가 생기면 명령어 한 번으로 팩이 만들어지고, 연구수당·간접비·위탁비 등의 허용 상한이 편성액 기준으로 자동 계산됩니다. 근거를 누르면 지침 원문 인용 위치가 하이라이트됩니다.

### 10. (오후) 근거 원문 하이라이트 개선
- **무엇을:** 원문 미리보기에서 빨간색으로 칠하는 검색어를 "추측 단어" 대신 확실한 근거(원문 인용 · 출처 조문 "제8조·제23조" · 규정 속 수치 "3천만원", "6개월")로 바꿨고, 못 찾으면 항목명으로 2차 검색합니다. 하이라이트 위치를 "근거 위치 1/3, 이전/다음" 버튼으로 오가는 내비게이션도 붙였습니다.
- **왜:** "연구개발계획서에" 같은 엉뚱한 단어가 칠해지거나, 근거를 하나도 못 찾아도 아무 안내가 없었습니다.
- **결과:** 규정의 핵심 수치·조문이 정확히 표시되고, 몇 군데에서 발견됐는지, 지금 몇 번째를 보는지 알 수 있습니다.

## 🔧 수정한 부분

- **(오후) 수정 전:** AI 추출 팩에서 연구수당 허용 상한이 0원으로, 간접비 상한은 현물을 빼지 않아 과대하게, 어떤 팩은 모든 비목이 "제한 없음"으로 표시됐습니다.
- **수정 이유:** 상한 계산기가 ① 인건비를 고정 이름표(ID)로만 찾아 AI 팩의 비목을 못 찾았고, ② "현물 제외" 문구를 반영하지 않았고, ③ 규칙이 비목에 연결되지 못한 채 저장되면 통째로 무시했기 때문입니다. "연구시설·장비비"(가운뎃점)와 "연구시설.장비비"(마침표) 같은 표기 차이도 연결을 끊었습니다.
- **수정 후:** 비목을 이름으로도 찾고(표기 차이 무시), 현물은 "제외" 명시 시에만 차감("포함"이면 유지)하며, 연결이 끊긴 규칙은 화면 계산 시점에 이름으로 재매칭합니다. 이미 적용해 둔 팩도 재추출 없이 바로 복구됩니다.

- **수정 전:** 문서를 올릴 때 "다른 사용자와도 공유" 체크박스를 켰는데도, 관리자 승인 화면에 아무것도 뜨지 않는 문제가 있었습니다.
- **수정 이유:** 파일을 고르는 순간 업로드가 바로 시작되는데, 그 처리 도중 화면 코드가 파일 선택창을 미리 비워버렸습니다. 그 뒤에 "공유 신청" 단계에서 파일 목록을 다시 읽다 보니 이미 빈 목록을 읽게 됐고, 보낼 파일이 없으니 에러도 없이 그냥 아무 일도 일어나지 않았습니다.
- **수정 후:** 파일 목록을 맨 처음에 딱 한 번만 복사해두고, 그 복사본을 끝까지 사용하도록 고쳤습니다. 이제 화면이 나중에 입력창을 초기화해도 영향받지 않습니다.

## 😵 어려웠던 점 & 해결

- **문제:** "분명 공유 체크하고 업로드했는데 관리자 화면엔 안 보인다"는 제보를 받았는데, 에러 메시지가 하나도 없어서 코드만 봐서는 원인을 짐작하기 어려웠습니다.
- **원인:** Supabase(데이터베이스 서비스) 화면에서 실제 신청 기록 테이블을 직접 조회해보니, 신청 기록 자체가 아예 0건이었습니다. 즉 "승인이 안 된" 게 아니라 "애초에 신청이 접수되지도 않은" 상황이라는 걸 먼저 확인할 수 있었습니다.
- **해결:** 이 사실을 단서로 코드를 다시 훑어보니 위 "수정한 부분"에 적은 파일 목록 초기화 순서 문제를 찾아냈습니다. 화면 코드만 읽어서는 못 찾았을 문제를, 실제 데이터베이스 상태를 먼저 확인하는 방식으로 정확히 짚어냈습니다.

- **(오후) 문제:** 예산 편성 슬라이더를 하나 움직이면 손대지 않은 다른 비목의 슬라이더까지 같이 움직여 보였습니다.
- **원인:** 각 슬라이더의 눈금 최대값을 "그 비목 금액 + 남은 잔액"으로 잡았는데, 한 비목을 조절하면 잔액이 변하면서 다른 슬라이더들의 눈금 기준이 통째로 바뀌었습니다. 금액은 그대로인데 눈금이 바뀌니 손잡이 위치가 움직여 보인 것입니다.
- **해결:** 눈금을 모든 슬라이더에서 "총사업비"로 고정하고(손잡이 위치 = 총사업비 대비 비율), 잔액·상한 제한은 드래그한 값에만 적용하도록 바꿨습니다. 이제 다른 슬라이더는 미동도 하지 않습니다.
