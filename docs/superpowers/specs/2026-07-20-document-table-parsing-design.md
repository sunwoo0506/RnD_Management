# 문서 파싱 — 표(테이블) 구조 보존 설계

## 배경 / 문제

과제온의 핵심 가치는 사용자가 업로드한 공고문·지침(HWP/HWPX/PDF)을 AI가 읽어서 그 사업에 맞는 예산 편성·근거 규정을 자동으로 만들어주는 것이다. 이 파이프라인의 첫 단계인 `src/extract.ts`의 문서 텍스트 추출이 정확하지 않으면, 이후 AI 추출·비목 연결·상한 계산이 전부 잘못된 데이터 위에서 동작하게 된다.

현재 세 파서(HWP, HWPX, PDF) 모두 **표(테이블)의 행·열 구조를 보존하지 않고**, 표 안의 텍스트를 일반 문단처럼 순서대로 이어붙인다. 실제 공고문의 예산 항목 표(연차별 정부지원금·기관부담금 등)는 **병합 셀이 흔해서**, 구조 정보 없이 텍스트만 이어붙이면 AI가 어떤 숫자가 어느 항목·연차에 속하는지 잘못 해석하기 쉽다. 이는 실제로 관찰된 오작동(예: "위탁연구개발비" 규정이 "연구수당"·"연구실운영비"에 잘못 연결됨, 허용 상한이 0원으로 잘못 계산됨)의 근본 원인 중 하나로 지목되었다.

**범위**: 이 스펙은 표 구조 보존에만 집중한다. 스캔본 PDF의 OCR 자동 처리는 별도 스펙으로 다룬다 (범위 밖).

## 목표

- HWP, HWPX, PDF 세 형식 모두에서, 표 안의 셀 값이 원래 어느 행·열(및 병합된 상위 항목)에 속하는지 잃지 않고 텍스트로 추출한다.
- 병합 셀(rowSpan/colSpan)을 정확히 처리한다 — 실제 예산표에 흔하므로 이번 스펙의 핵심 요구사항이다.
- 표가 아닌 일반 문단은 기존 동작을 그대로 유지한다 (회귀 없음).
- 표 파싱이 실패해도 문서 전체 추출이 실패하지 않는다 (표 단위 격리).
- `extractDocumentText()`의 외부 인터페이스(`{ text, method }`)는 변경하지 않는다 — 호출부(SetupWizard, DocUpdatePanel, Edge Function)는 수정 불필요.

## 비목표

- 스캔본(이미지 기반) PDF의 OCR 자동 처리 — 별도 스펙.
- 표 안에 또 다른 표가 중첩된 경우의 완벽한 처리 — 발견 시 표 단위 폴백(구조 없는 텍스트)으로 안전하게 저하.
- PDF의 다단(multi-column) 일반 레이아웃을 표로 오인하지 않도록 하는 완벽한 보장 — 휴리스틱의 한계로 남긴다 (아래 리스크 참고).

## 설계

### 출력 형식 — 공통 마크다운 표

세 파서가 표를 만나면 공통 헬퍼 `renderMarkdownTable(rows: string[][]): string`(신규, `src/extract.ts`)을 통해 마크다운 표(`| 셀 | 셀 |` + 헤더 구분선)로 렌더링한다. 표가 아닌 부분은 지금처럼 문단 텍스트로 남고, 문서에 나온 순서 그대로 문단 텍스트와 마크다운 표가 섞여서 하나의 `text` 문자열을 이룬다.

마크다운을 선택한 이유: LLM(현재 OpenAI `gpt-5.6-sol` 기반 Edge Function)이 가장 안정적으로 이해하는 구조화 텍스트 형식이며, Edge Function의 추출 프롬프트를 수정하지 않아도 그대로 활용 가능하다.

### 병합 셀 처리 원칙 (HWPX·HWP 공통)

각 셀은 좌표(행 주소, 열 주소)와 스팬(행·열이 몇 칸에 걸쳐 병합됐는지) 정보를 갖는다. 이 정보로 **완전한 격자(2차원 배열)**를 만들고, 병합된 셀의 값은 그 셀이 덮는 **모든 칸에 복제**해 넣는다. 예: "기관부담연구개발비"가 3칸에 걸쳐 병합돼 있으면 격자의 3칸 모두에 같은 값이 들어가고, 그 아래 "현금/현물/합계" 하위 헤더 행이 이어진다. 결과를 그대로 마크다운으로 펴내면 2줄짜리 헤더처럼 보이지만, 각 열이 어떤 상위 항목에 속하는지 정보는 손실 없이 보존된다.

### HWPX (OWPML XML)

- `<hp:tbl>...</hp:tbl>` 블록을 식별해 표 영역을 분리한다.
- 블록 내 각 `<hp:tc>`(셀)의 `cellAddr`(좌표)·`cellSpan`(병합 크기)을 읽어 격자에 배치한다 (문서 순서가 아니라 좌표 기준 배치 — 병합으로 인한 순서 밀림에 안전).
- 셀 텍스트는 기존 문단(`<hp:t>`) 추출 로직을 재사용한다.
- 표가 아닌 영역은 현재의 평면 토큰 스캔을 그대로 사용한다.

### HWP (5.0 바이너리)

- 현재 `decodeHwpParagraphText`가 건너뛰기만 하던 인라인 컨트롤 객체 표시 중, 표(`tbl `) 컨트롤을 식별한다.
- 표 컨트롤을 만나면 뒤따르는 표 레코드에서 전체 행·열 개수를 읽고, 각 셀 레코드에서 좌표·병합 크기를 읽는다.
- 각 셀에 속한 문단(PARA_TEXT) 레코드를 재귀적으로 수집해 셀 텍스트를 구성한다 — 이를 위해 현재 평면(단일 단계) 레코드 워커를 "현재 중첩 단계"를 추적하는 구조로 확장한다.
- HWPX와 동일한 격자 조립 + 병합 복제 로직, 동일한 `renderMarkdownTable`을 사용한다.
- 세 형식 중 구조 정보가 레코드에 흩어져 있어 가장 손이 많이 가고 리스크가 크다 — 아래 안전장치가 특히 중요하다.

### PDF

- pdfjs `getTextContent()`의 각 텍스트 조각이 가진 좌표(x, y)를 활용한다 (현재는 버려짐).
- y좌표로 텍스트 조각을 행으로 묶고, 각 행 안에서 x좌표 정렬 후 조각 간 간격이 유난히 벌어진 지점을 열 경계로 판단한다.
- 연속된 여러 행에서 열 경계 위치가 비슷하게 반복되면 표 영역으로 판단해 마크다운 표로 렌더링한다.
- 표로 판단되지 않으면 기존처럼 공백으로 이어붙이는 방식을 그대로 유지한다 — 오탐(false positive) 시에도 기존 동작으로 안전하게 저하된다.
- HWPX·HWP와 달리 병합 셀 정보가 명시적이지 않고 좌표 추정에 의존하므로, 표가 매우 촘촘하거나 다단 레이아웃인 경우 완벽하지 않을 수 있다 (알려진 한계).

### 오류 처리 / 안전장치

- **표 단위 격리**: 표 하나의 파싱이 실패(예상 밖 구조, 병합 정보 불일치 등)해도 그 표만 현재처럼 구조 없는 텍스트로 폴백하고, 문서의 나머지 부분과 다른 표는 정상 처리를 계속한다. 문서 전체 추출이 실패하는 일은 없다.
- **격자 정합성 검증**: 좌표·병합 정보로 격자를 만들었을 때 셀이 겹치거나 행 길이가 맞지 않는 등 이상이 발견되면, 억지로 끼워맞추지 않고 그 표를 구조 없는 텍스트로 폴백한다.
- 기존 상위 안전장치(HWP 서명·암호화 검사, 텍스트 없는 PDF 감지 등)는 그대로 유지한다.

### 인터페이스 영향

없음. `extractDocumentText(file): Promise<ExtractedDoc>`의 시그니처와 반환 형태(`{ text: string, method }`)가 그대로 유지되므로, `SetupWizard.tsx`·`DocUpdatePanel`(App.tsx)·Edge Function 프롬프트 어느 쪽도 수정이 필요 없다.

## 테스트 전략

`src/extract.test.ts`의 기존 단위 테스트 스타일(HWP/HWPX 헬퍼 함수를 직접 호출)을 따른다.

- **HWPX**: `cellSpan`이 있는 표 XML을 직접 구성해, 병합 셀 값이 격자의 모든 해당 칸에 올바르게 복제되는지 검증.
- **HWP**: 표 컨트롤 + 셀 레코드를 손으로 구성한 바이너리 섹션으로, 좌표·병합이 격자에 올바르게 반영되는지 검증.
- **PDF**: pdfjs 텍스트 조각(좌표 포함)을 흉내 낸 목(mock) 데이터로, 행/열 인식과 마크다운 표 변환을 검증.
- 세 형식 모두 "표가 아닌 일반 문단은 기존 동작과 동일하게 나오는지"를 회귀 테스트로 포함한다.

## 리스크 / 알려진 한계

- **HWP 바이너리 파싱이 가장 리스크가 크다** — 문서화가 불완전하고 예외 케이스가 많은 영역이라, 실제 다양한 공고문 샘플로 충분히 검증하지 않으면 특정 문서에서 조용히 폴백(구조 없는 텍스트)될 수 있다. 안전장치 덕분에 "지금보다 나빠지지 않음"은 보장되지만 "항상 개선됨"은 보장되지 않는다.
- **표 중첩**(표 안에 표)은 지원 범위 밖이며, 발견 시 표 단위 폴백으로 처리된다.
- **PDF 표 탐지는 휴리스틱**이라 다단 레이아웃을 표로 오인하거나, 매우 촘촘한 표를 놓칠 가능성이 있다. 발생 빈도는 실제 사용 중 관찰하며 추후 조정한다.

## 후속 스펙 (범위 밖)

- 스캔본 PDF의 OCR 자동 처리 (이미지 렌더링 → tesseract 또는 vision LLM 폴백).
