# 집행·증빙 화면 재설계 기획안

**작성일:** 2026-07-22 (rev.2 — 결정사항 반영)
**대상:** `src/App.tsx`의 `Spending` 컴포넌트(1239행~) 전면 개편
**상태:** 기획 — 구현 계획 수립 전 최종 검토

---

## 1. 지금 무엇이 문제인가

현재 집행·증빙 화면은 **집행건 등록 폼이 곧바로 뜨고**, 예산을 얼마나 썼는지는 폼 안의 작은 잔액 미리보기(`balance-preview`)로만 보인다. 그래서

- **전체 그림이 없다.** 어느 비목에 얼마 남았는지 보려면 비목을 하나씩 바꿔가며 폼을 봐야 한다.
- **시간축이 없다.** 사업기간 중 언제 얼마를 써야 하는지, 지금 진도가 빠른지 느린지 알 수 없다.
- **편성과 집행이 끊겨 있다.** 편성 화면에서 연구활동비를 `회의비 800만 / 출장비 1,200만`으로 나눠놨는데, 집행 화면은 그 세목을 모른다. 비목 단위로만 집행해서 **세목별로 계획대로 쓰고 있는지 확인할 수 없다.**
- **입력 항목이 모든 비목에 똑같다.** 회의비인데 회의 목적·장소·참석자를 적을 칸이 없다.

## 2. 새 화면 구성

```
┌──────────────────────────────────────────┬─────────────────────────────┐
│ A. 예산 집행 대시보드                     │ B. 월별 집행계획 · 집행금액   │
│    (확정 예산 · 집행 · 잔액)              │    (▶ 접었다 펼치는 토글)     │
├──────────────────────────────────────────┴─────────────────────────────┤
│ C. 집행건 등록                                                          │
│    ① 비목·세목 선택  →  ② 유의사항·필요 증빙  →  ③ 상세 입력           │
├─────────────────────────────────────────────────────────────────────────┤
│ D. 집행 내역 목록 (기존 유지)                                            │
└─────────────────────────────────────────────────────────────────────────┘
```

### A. 예산 집행 대시보드 (상단 좌)

편성 화면에서 확정된 예산(`project.budgets`, `visibleCategories`로 0원 비목 제외)을 **비목 행 + 세목 하위 행**으로 보여준다.

| 비목 · 세목 | 예산 | 집행금액 | 잔액 | 소진율 |
|---|---:|---:|---:|---|
| **인건비** | 50,000,000 | 12,000,000 | 38,000,000 | ▓▓░░░ 24% |
| **연구활동비** | 30,000,000 | 8,500,000 | 21,500,000 | ▓▓▓░░ 28% |
| &nbsp;&nbsp;└ 회의비 | 8,000,000 | 3,500,000 | 4,500,000 | 44% |
| &nbsp;&nbsp;└ 출장비 | 12,000,000 | 5,000,000 | 7,000,000 | 42% |
| **합계** | **100,000,000** | **20,500,000** | **79,500,000** | 21% |

- **집행금액** = 집행건들의 `amount`(공급가액, 부가세 제외) 합계. 기존 집계 기준(`Spending`의 `spent`)과 동일.
- 세목 행은 비목 행을 눌러 펼친다. 세목이 편성되지 않은 비목은 하위 행이 없다.
- 행을 누르면 그 비목·세목으로 아래 등록 폼이 맞춰진다.

### B. 월별 집행계획 · 집행금액 (상단 우, 토글)

`<details>` 토글. 기본 접힘. **A에서 고른 비목·세목 기준**으로 표시한다(아무것도 안 고르면 전체 합계).

| 월 | 계획 | 집행 | 차이 |
|---|---:|---:|---:|
| 2026-07 | 666,667 | 4,200,000 | ▲ 3,533,333 |
| 2026-08 | 666,667 | 0 | ▼ 666,667 |
| … | | | |
| **합계** | **8,000,000** | **3,500,000** | |

**계획 — 자동 계산 + 직접 수정 가능**

- 기본값 = `해당 비목·세목 예산 ÷ 사업기간 월수`. 월수는 기존 `monthsBetween(project.startDate, project.endDate)`(`src/rules.ts:781`).
- 나누어떨어지지 않는 나머지는 **마지막 달에 몰아준다.** 그래야 계획 합계가 예산과 정확히 일치한다.
- **사용자가 각 월 칸을 직접 고칠 수 있다.** 실제 과제는 장비를 초반에 사는 식으로 앞뒤가 쏠리기 때문이다.
- 고친 칸만 저장한다(§5). 안 고친 칸은 계속 자동 계산값을 따라가므로, **예산이 바뀌면 안 고친 칸은 자동으로 재조정된다.**
- 고친 결과 계획 합계가 예산과 어긋나면 표 하단에 `계획 합계 82,000,000 / 예산 80,000,000 · ▲2,000,000 초과`처럼 **드러내되 막지는 않는다.** (편성 중간 상태에서 저장을 막으면 못 쓴다)

**집행**

- 그 달에 집행일(`expense.date`)이 속한 집행건의 `amount` 합계.
- **집행 합계는 A 대시보드의 집행금액과 같다.** 같은 데이터를 다르게 묶는 것뿐이다.
- ⚠️ **단, 사업기간 밖 집행일을 가진 건이 있으면 어긋난다.** 표 맨 아래 `기간 외` 행으로 드러낸다 — 정산 때 문제가 되는 건이라 숨기면 안 된다.

### C. 집행건 등록 — 비목을 가장 먼저

```
① 무엇을 집행하나요?
   [ 비목 ▾ ]  →  [ 세목 ▾ ]        ← 가장 먼저, 크게

② 이 비목은 이렇게 씁니다              ← 고르면 나타남
   ⚠ 유의사항          (rulesFor kind='warning')
   📋 필요한 증빙       (primaryEvidence — 1순위 규정 > 2순위 상위규정 > 3순위 앱 기본)
   💰 회의비 잔액 4,500,000원 · 이번 달 계획 666,667원 중 0원 집행

③ 집행 내용
   집행일 · 결제수단 · 공급가액 · 부가세액 · 합계금액 · 용도 · 거래처
   + 세목별 추가 항목 (회의비면 회의 목적·장소·참석자 …)
```

②는 지금도 있는 것들이지만 **폼 하단이 아니라 비목 바로 아래로 올린다.** 고르자마자 "이건 이렇게 써야 한다"를 읽고 나서 금액을 적게 하는 순서다.

## 3. 비목·세목 선택 — 편성한 것만 고른다

### 핵심 원칙

> **집행 화면은 새 선택지를 만들지 않는다. 예산편성 화면에서 이미 정한 세목만 고르게 한다.**

규정팩의 전체 목록(연구활동비 `allowedItems` 71개)을 펼치는 것이 아니라, `project.budgets[].subItems`에 **실제로 편성된 세목만** 올린다. 편성 화면(`App.tsx:960~980`)에서 `+ 세목 나누기`로 이미 골라둔 것들이다.

| 단계 | 출처 | 예시 | 예산 차감 |
|---|---|---|---|
| **비목** | `visibleCategories(pack, project)` | 인건비, 연구활동비 | ✅ **여기서 차감** |
| **세목** | `project.budgets[].subItems` | 회의비, 출장비 | ❌ (집계·계획 단위) |

3단 구조를 2단으로 줄였다. **인정 항목(`allowedItems`) 선택 단계는 없앤다** — 편성 세목이 이미 그 역할을 한다.

### 세목 필수 규칙

**세목 필수는 조건부 규칙이다** (§7-① 결정 A).

- 세목이 1개 이상 편성된 비목 → 세목 선택 **필수**. 회의비·출장비는 규정이 요구하는 증빙이 서로 완전히 다르기 때문이다.
- 세목을 안 나눈 비목(인건비 등) → 세목 칸을 **숨기고** 비목만으로 등록한다.

편성 화면의 세목 나누기는 선택 기능이므로(`subItems`가 `optional`), 집행 화면은 **편성된 만큼만 요구한다.**

### 세목 → 규정 연결

세목 이름으로 `pack.referenceCategories`에서 기준을 찾는다. 이미 있는 `matchingCategory`·`referenceStandardFor`(`src/rules.ts:499`, `:533`)가 하는 일이다.

실제 데이터 확인 결과 (`nrd2026-forprofit`) — **규정DB를 새로 채울 필요가 없다:**

| 세목 | 증빙 규칙 | 인정 항목 | 상한 |
|---|---|---|---|
| 회의비 | 10만원 초과 기본증빙 / 10만원 이하 간소화 / 사용 입증자료 | 5개 | 별도 정률 총액상한 없음 |
| 출장비 | 국외출장 계획서(사전) / 결과보고서(사후) / 사용 입증자료 | 9개 | 기관 여비규정 기준 |

**이름이 한 단계 아래일 때의 폴백:** 편성 세목 이름이 `회의 식비`처럼 인정 항목 수준일 수도 있다(`subItemOptions`가 없는 팩은 `allowedItems`가 세목 후보가 된다 — `subItemChoicesFor`, `src/rules.ts:563`). 이때는 **그 이름을 `allowedItems`에 포함하는 상위 세목을 역으로 찾는다.** `회의 식비` → `회의비`. 역방향 색인 하나면 된다.

## 4. 세목별 추가 입력 항목

### 구조 — 명세 표 하나로 정의

새 세목이 생겨도 타입을 안 고치도록, 입력 칸 구성을 **코드 한 곳의 표**로 둔다 (`src/spendingForms.ts` 신설).

  hint?: string;

export const DETAIL_FIELDS: Record<string, DetailField[]> = {

  ],

    { key: 'traveler',   label: '출장자',   required: true, hint: '쉼표로 구분' },

### 문서 생성은 "확장 가능하게"만 준비

지금 `downloadTemplate`(`src/exporters.ts:96`)은 **빈 양식**만 만든다. 나중에 입력값으로 채워진 회의록·출장신청서·출장보고서를 만들 수 있도록, **이번에는 자리만 잡아둔다.**

```ts
// 세목 → 이 집행건에서 만들 수 있는 문서. 이번 단계에서는 목록만 쓰고,
// 실제 생성(fill)은 다음 단계에서 exporters.ts에 붙인다.
export const DETAIL_DOCUMENTS: Record<string, string[]> = {
  회의비: ['회의록'],
  출장비: ['출장신청서', '출장보고서'],
};
```

- 필드 `key`는 **문서 서식의 칸 이름과 1:1로 대응**하도록 지금 정해둔다. 나중에 `fillTemplate(type, expense.details)` 한 줄이면 붙는다.
- 이번 범위에서는 **입력·저장·표시까지만.** 집행건 카드에 `[회의록 Word]` 버튼은 다음 단계.

## 5. 데이터 모델 변경

```ts
// src/types.ts

interface Expense {
  // …기존
  subItemId?: string;                // 편성 세목 연결 (BudgetSubItem.id) — 집계·월별 계획의 키
  subItemName?: string;              // 세목 이름 스냅샷 — 규정 조회·표시용. 세목이 삭제돼도 남는다
  details?: Record<string, string>;  // 세목별 추가 입력 (DETAIL_FIELDS의 key → 값)
}

// 월별 집행계획에서 사용자가 직접 고친 칸만 저장한다.
// 없는 칸은 (예산 ÷ 월수) 자동 계산값을 쓰므로, 예산이 바뀌면 자동으로 따라간다.
interface MonthlyPlanEntry {
  categoryId: BudgetCategoryId;
  subItemId?: string;   // 없으면 비목 전체 기준
  month: string;        // 'YYYY-MM'
  amount: number;
}

interface Project {
  // …기존
  monthlyPlan?: MonthlyPlanEntry[];
}
```

**설계 원칙 세 가지**

1. **예산·잔액은 `categoryId`로만 계산한다.** 세목은 집계·계획·규정조회의 단위일 뿐, 예산 변경 관리(`ChangeManagement`)·정산이 보는 돈의 단위는 비목이다.
2. **세목 계획의 합 = 비목 계획.** 편성 화면의 기존 규칙(`"세목이 있으면 비목 금액은 항상 세목 합계"`, `App.tsx:963`)과 같은 원칙을 월별 계획에도 적용한다.
3. **`subItemId`(연결)와 `subItemName`(스냅샷)을 둘 다 저장한다.** 편성에서 세목을 지워도 집행 이력이 무엇이었는지 남아야 한다.

기존 집행건 마이그레이션은 **불필요**(⑤ — 아직 실사용 데이터 없음). 그래도 전 필드 `optional`이라 예전 데이터가 있어도 열린다.

| 단계 | 내용 | 위험도 |
|---|---|---|
| **1** | 대시보드(A) — 비목·세목 집계 순수 함수 + 표 | 낮음 (읽기 전용) |

계산 로직은 전부 **화면 없이 테스트되는 순수 함수**로 `src/spending.ts`(신설)에 두고 `spending.test.ts`에서 검증한다.

## 7. 남은 결정 사항

### ① 세목을 안 나눈 비목은 어떻게 하나 — **결정: A**

**세목이 편성되지 않은 비목은 비목만으로 집행한다.** 집행 화면은 막지 않고, 편성 화면에도 경고를 붙이지 않는다.

따라서 "세목 필수"는 **조건부 규칙**이다.

| 편성 상태 | 집행 화면 |
|---|---|
| 세목이 1개 이상 편성됨 | 세목 선택 **필수** |
| 세목 없음 | 세목 칸 자체를 **숨김** — 비목만으로 등록 |

세목을 안 나눈 비목은 대시보드에 하위 행이 없고, 월별 계획도 비목 단위로만 잡힌다. 나중에 편성에서 세목을 나누면 그때부터 세목 단위로 갈라진다.

### ② 월별 계획을 고친 뒤 예산이 바뀌면?

§2B대로 **고친 칸은 그대로 두고, 안 고친 칸만 재계산**한다. 합계가 어긋나면 표시만 하고 막지 않는다.
→ 이대로 진행해도 되는지 확인만 필요하다.

### ③ 세목을 편성에서 삭제하면 그 집행건은?

`subItemName` 스냅샷이 남아 `회의비(삭제된 세목)`으로 표시된다. 집행금액은 **비목 집계에는 계속 포함**되고 세목 행에서만 빠진다.
→ 이대로 진행해도 되는지 확인만 필요하다.
