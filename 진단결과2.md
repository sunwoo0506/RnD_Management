# React 버전 프로젝트 진단 결과

**진단일:** 2026-07-16  
**대상:** `src/`, React/Vite 설정, 브라우저 저장소, 테스트 및 프로덕션 빌드  
**제외:** `app.py`, Streamlit 구현 및 Streamlit 배포 방식

## 1. 결론

현재 React 앱은 **데모 가능한 기능형 MVP**다. 과제 등록, 예산 초안, 상한 경고, 집행 등록, 증빙 파일 저장, 예산 변경, Excel·Word·PDF 내보내기가 하나의 흐름으로 연결되어 있고 테스트와 프로덕션 빌드도 성공한다.

다만 아직 **실사용 가능한 정산 도구**로 보기는 어렵다. 가장 큰 이유는 규정값의 검증 근거가 코드에 없고, 일부 변경 작업이 상한을 위반해도 저장되며, 데이터가 한 브라우저에만 남고, 모바일 화면이 사실상 지원되지 않기 때문이다.

**현재 판정: 방향 명확 + 기능형 MVP 완성 + 공개 검증 전 안정화 필요**

## 3. 잘된 점

### 핵심 사용자 흐름이 실제로 연결되어 있다

과제 생성 → 예산 자동 배분 → 비목별 상한 표시 → 집행 등록 → 증빙 체크리스트 → 비목 변경 → 문서 내보내기의 흐름이 구현되어 있다. 화면만 있는 목업이 아니라 입력값이 상태와 결과에 반영된다.

### 브라우저 저장 방식이 MVP에 적절하다

- 구조화된 과제 데이터는 `localStorage`에 저장된다.
- 증빙 파일은 용량이 더 큰 `IndexedDB`에 저장된다.
- 새로고침 후에도 입력 데이터와 파일을 유지할 수 있다.

로그인과 서버 없이 한 명이 같은 기기에서 검증하는 단계에는 합리적인 선택이다.

### 타입 검사와 자동 테스트 기반이 있다

TypeScript의 `strict` 옵션이 켜져 있고, 핵심 흐름을 검사하는 테스트가 있다. 현재 테스트는 예산 초안, 상한 경고, 집행·증빙 생성, 참여율 경고, 예산 변경 문서 버튼을 확인한다.

### 내보내기 기능이 구체적이다

예산표 Excel, 변경 비교표 Word/PDF, 공문 Word, 증빙 템플릿 Word가 구현되어 있다. 사용자가 결과물을 업무에 옮겨 쓰기 쉬운 형태라는 점은 서비스 강점이다.

## 4. 우선 수정이 필요한 문제

### P0 — 규정 데이터의 정확성과 출처를 확인할 수 없다

`src/rules.ts`에는 비목별 상한과 증빙 요건이 고정값으로 들어 있다. 하지만 어느 공고·협약·규정의 몇 조항을 기준으로 했는지, 어떤 사업 유형에 적용되는지 표시되지 않는다. 정부과제 규정은 사업과 협약 조건에 따라 달라질 수 있으므로 현재 값을 보편적인 중기부 기준처럼 사용하면 잘못된 안내가 될 위험이 있다.

**권장 조치:** 각 규칙에 `sourceTitle`, `sourceUrl`, `effectiveDate`, `scope`, `verifiedAt`을 추가하고, 검증 전 값은 화면에서 “예시 기준”으로 명확히 표시한다.

### P0 — 예산 변경 후 받는 비목의 상한을 검사하지 않는다

예산 변경은 보내는 비목의 잔액과 양수 여부만 확인한다. 받는 비목이 변경 후 허용 상한을 넘는지는 막지 않는다. 따라서 예산 편성 화면에서는 “상한 초과”로 표시하면서 변경 저장 자체는 허용하는 모순이 생긴다.

**권장 조치:** 변경 전 저장 단계에서 대상 비목의 변경 후 비율을 계산하고, 상한 초과 시 저장을 차단하거나 승인 필요 상태로 분리한다. 이 조건을 자동 테스트에도 추가한다.

### P0 — 프로젝트 초기화 뒤 증빙 파일이 남는다

“과제 제거”는 `localStorage`의 과제 데이터만 지우고 IndexedDB의 증빙 파일은 삭제하지 않는다. 사용자는 파일에 다시 접근할 수 없지만 브라우저 저장 공간에는 개인정보성 자료가 고아 데이터로 계속 남는다.

**권장 조치:** 초기화 시 현재 과제에 연결된 증빙 파일을 함께 삭제할지 선택하게 하고, 전체 삭제 함수와 테스트를 추가한다.

### P1 — 모바일과 작은 노트북 화면을 지원하지 않는다

`body`에 `min-width: 1180px`이 지정되어 있고 반응형 규칙도 1250px 이하에서 여백만 줄인다. 휴대폰에서는 가로 스크롤이 생기며 고정 사이드바와 표를 정상적으로 사용하기 어렵다. 또한 7~10px 크기의 본문 텍스트가 다수라 가독성과 접근성이 낮다.

**권장 조치:** 768px 전후에서 사이드바를 상단 또는 드로어 내비게이션으로 바꾸고, 표를 카드형 또는 가로 스크롤 영역으로 전환한다. 일반 본문은 최소 14px 수준으로 조정한다.

### P1 — 브라우저 하나만 사용하는 구조인데 “계정” 표현이 혼동을 준다

두 번째 사용자를 등록해도 인증, 초대, 동기화가 일어나지 않는다. 같은 브라우저 데이터에 이름과 이메일만 추가된다. 현재의 “계정 사용자”, “담당자 초대”라는 표현은 실제 공동 사용 기능으로 오해될 수 있다.

**권장 조치:** MVP에서는 “담당자 정보”로 이름을 바꾸거나, 실제 계정·동기화를 구현할 때까지 초대 표현을 제거한다.

### P1 — 저장 실패와 데이터 버전 변경에 취약하다

`localStorage` 저장 실패는 처리하지 않고, 불러온 JSON이 현재 `Project` 구조와 맞는지 검증하지 않는다. 저장 공간 부족, 손상된 값, 다음 버전의 필드 변경이 생기면 조용히 데이터가 사라지거나 화면 오류가 날 수 있다.

**권장 조치:** 저장 오류 안내, 런타임 스키마 검증, 데이터 버전과 마이그레이션을 추가한다. 최소한 백업 JSON 내보내기·가져오기를 제공하면 사용자 데이터 손실 위험을 낮출 수 있다.

### P1 — PDF가 한국어 문서로 출력되지 않는다

변경 비교표 PDF는 영문 제목과 영문 category id를 사용하며, 프로젝트명의 한글도 제거한다. 한국어 폰트를 포함하지 않아 실제 업무 문서 품질이 Word 출력보다 낮다.

**권장 조치:** 한글 폰트를 포함해 한글 제목·비목명·변경 사유를 출력하거나, 품질을 확보하기 전에는 PDF 기능을 숨기고 Word만 제공한다.

## 5. 개선하면 좋은 문제

### 번들 크기가 크다

문서 생성 라이브러리들이 첫 화면 번들에 함께 포함되어 메인 청크가 1MB를 넘는다. 초기 로딩에 필요하지 않은 Excel·Word·PDF 코드를 버튼 클릭 시 동적 import하도록 분리하면 초기 다운로드를 줄일 수 있다.

### `App.tsx`에 역할이 집중되어 있다

약 200줄 안에 모든 화면, 상태 변경, 파일 처리, 알림 로직이 모여 있다. 현재 규모에서는 동작하지만 기능을 추가할수록 수정 영향 범위가 커진다.

**권장 분리:** `pages/`, `components/`, `hooks/useProject.ts`, `services/storage.ts`, `services/exporters.ts` 구조로 단계적으로 나눈다.

### 변경 이력이 최신 1건만 남는다

실제 변경 관리에서는 누가 언제 무엇을 바꿨는지 누적 기록이 중요하다. 현재는 `latestChange` 하나만 저장해 다음 변경 때 이전 기록이 사라진다.

**권장 조치:** `latestChange`를 `changes: BudgetChange[]`로 바꾸고 삭제하지 않는 이력 목록을 제공한다.

### 테스트 범위가 성공 경로 중심이다

현재 7개 테스트는 좋은 출발점이지만 다음 항목은 빠져 있다.

- 받는 비목 상한 초과 차단
- 집행액 0·음수·NaN 방지
- 저장 공간 부족 및 손상 데이터 복구
- IndexedDB 업로드 실패와 초기화 시 파일 삭제
- PDF·Word·Excel 생성 실패
- 모바일 화면과 키보드 접근성
- 날짜 역전(종료일이 시작일보다 빠른 경우)

## 6. 유지할 것

- React + TypeScript + Vite 단일 프런트엔드 구조
- `localStorage` + IndexedDB 조합(단일 사용자 MVP 기간)
- 예산 → 집행 → 증빙으로 이어지는 핵심 흐름
- 규칙과 UI를 분리한 `rules.ts`
- 내보내기 로직을 분리한 `exporters.ts`
- Vitest와 Testing Library 기반 자동 테스트

## 7. 지금은 미룰 것

- 로그인과 서버 DB
- 실제 이메일 발송
- 복잡한 권한 관리
- 여러 과제 동시 관리
- 실시간 공동 편집

위 기능은 공개 링크로 5명 안팎의 사용자가 핵심 흐름을 끝까지 수행한 뒤 요구가 확인되면 추가하는 편이 낫다.

## 8. 권장 작업 순서

1. 규정값의 공식 출처와 적용 범위를 검증하고 화면에 근거를 표시한다.
2. 예산 변경 상한 검사와 날짜·금액 검증을 추가한다.
3. 초기화 시 IndexedDB 증빙까지 안전하게 삭제한다.
4. 위 세 항목의 실패 경로 테스트를 추가한다.
5. 모바일 레이아웃과 글자 크기를 개선한다.
6. 내보내기 라이브러리를 동적 로딩해 번들 크기를 줄인다.
7. 정적 배포 후 실제 사용자에게 과제 등록부터 증빙 등록까지 수행하게 한다.

## 9. 이번 단계의 완료 기준

아래 조건을 만족하면 “React MVP 공개 검증 준비 완료”로 판단할 수 있다.

- 모든 규칙에 확인 가능한 출처와 적용 범위가 있다.
- 상한을 위반하는 예산 변경이 의도 없이 저장되지 않는다.
- 프로젝트 삭제 시 증빙 파일 처리 방식을 사용자가 이해하고 선택할 수 있다.
- 휴대폰에서 과제 등록과 집행 등록을 완료할 수 있다.
- 핵심 성공·실패 경로 테스트가 모두 통과한다.
- 배포 링크에서 새로고침 후 데이터 유지와 문서 다운로드가 정상 동작한다.

---

**최종 진단:** 기능 개수를 더 늘릴 단계가 아니라, 현재 핵심 기능의 규정 신뢰성·데이터 안전성·모바일 사용성을 보강한 뒤 실제 사용자 검증으로 넘어갈 단계다.
